# 技能本地化与极致打磨 (Skill Localization & Polishing)

## 概述

此技能强制执行从“普通文档”到“专家级行为准则”的升维过程。它要求在确保智能体**完全理解**意图和边界的前提下，剔除冗余噪音，实现“能力最大化，Token 消耗合理化”。

**核心原则：**
1.  **高信噪比 (High Signal-to-Noise)**：拒绝废话，但严禁为了省字而牺牲逻辑的完备性。每一个字都必须增加智能体的认知清晰度。
2.  **认知完备 (Cognitive Completeness)**：必须提供足够的上下文（Context）和原理（Rationale），确保智能体不仅知道“怎么做”，更理解“为什么”，从而具备处理边缘情况的判断力。
3.  **防御性极简 (Defensive Minimalism)**：只有在不引入歧义的前提下才进行简化。宁可多写一句明确的限制，也不留下一片模糊的空白。

## 何时使用

- 创建新的中文技能时。
- 优化现有技能，特别是当技能虽然简短但执行效果不佳（智能体经常误解）时。
- 用户要求提升技能的专业度、逻辑闭环能力或抗幻觉能力时。
- **用户请求批量处理多个技能时（如“检查所有技能”）。**

## 核心法则

### 1. 必须执行意图锚定 (Intent Anchoring)

在编写文档前，**必须**先进行深度思考，明确以下锚点：
1.  **核心痛点**：这个技能是为了解决什么具体的混乱或错误？（例如：智能体经常偷懒不写测试）。
2.  **能力边界**：这个技能能做什么，**绝对不能**做什么？（例如：UX 原型技能绝对不能用于生产代码）。
3.  **执行对象**：这是给初级执行者看的步骤表，还是给高级架构师看的原则书？

### 2. 必须启动双重自我博弈 (Dual Self-Play)

在定稿前，**必须**经历两轮对抗性长考：

*   **第一轮：傻瓜测试 (The Fool Test)**
    *   *扮演*：一个只会字面执行、不懂变通的初级智能体。
    *   *攻击*：如果你把这句话删了，我会不会因为缺乏上下文而不知所措？我会不会因为没有解释“为什么”而死板执行导致错误？
    *   *优化*：补全缺失的逻辑链路，确保“有理有据”。

*   **第二轮：律师测试 (The Lawyer Test)**
    *   *扮演*：一个试图钻空子、偷懒的狡猾智能体。
    *   *攻击*：我能不能利用这句话的模糊性来规避繁琐的工作？
    *   *优化*：将建议性语言升级为法律般的严密条款，封堵漏洞。

### 3. 必须实施高密度表达 (High Density Expression)

不追求绝对的短，但追求绝对的**精准**：
- **术语替代描述**：
    - ❌ *Low Density*: "请把现在的代码和之前的版本进行比对，看看哪里不一样。" (19字)
    - ✅ *High Density*: "执行 `git diff` 进行差异审计。" (8字，且信息量更大，包含了工具和动作属性)。
- **保留关键原理**：
    - ❌ *Too Simple*: "必须写测试。" (无法应对紧急情况的决策)
    - ✅ *Optimal*: "必须写测试。**原理**：无测试的代码是不可维护的技术债务，会阻断未来的重构路径。" (增加了决策权重，智能体更难找借口)。

### 4. 必须强制执行标准结构 (Structure Enforcement)

所有技能**必须**包含以下骨架：

| 部分 | 标题 | 要求 |
|------|------|------|
| Frontmatter | (无) | `name` + `description` (精准描述触发场景，而非功能)。 |
| Overview | 概述 | 核心定义 + 3大核心原则 (建立认知模型)。 |
| When to use | 何时使用 | 触发场景 + **何时不使用** (明确能力边界)。 |
| Core Rules | 核心法则 | **绝对命令**。每条法则应包含“动作”和必要的“原理/后果”。 |
| Excuses | 借口粉碎机 | 预判借口 vs 事实反击 (预先植入防御性思维)。 |
| Red Flags | 危险信号 | 触发“立即停止”的具体行为 (熔断机制)。 |

### 5. 必须实施批量任务原子化 (Batch Atomicity)

若任务涉及 **2 个及以上** 的技能文件修改或创建：
- **必须**先激活 `writing-plans` 技能。
- **必须**制定串行计划：计划中的每一个步骤（Task）**只能**专注于打磨 **1 个** 技能。
- **严禁**“一波流”：严禁试图在一个回复中生成或修改所有技能。
- **原理**：大模型的上下文窗口和注意力是稀缺资源。试图并行处理多个复杂逻辑文档会导致严重的“注意力弥散”和幻觉。逐个击破是保持高质量细节的唯一途径。

## 借口粉碎机 (Excuse Smasher)

| 借口 | 事实反击 |
|------|----------|
| “删掉所有解释性文字会让文档更简洁” | 盲目删除会丢失上下文。智能体需要理解“为什么”才能在复杂场景下正确决策。 |
| “我写的很清楚，智能体应该能懂” | 永远不要假设。必须通过“傻瓜测试”和“律师测试”来验证文档的鲁棒性。 |
| “这只是个小技能，不需要深度思考” | 垃圾进，垃圾出。平庸的技能会导致平庸的执行。 |
| “这就两三个文件，我直接一次性改完吧” | 批量修改会导致质量雪崩。为了防幻觉，必须强制拆解为原子步骤。 |

## 危险信号 (Red Flags)

如果出现以下情况，**立即停止**并重置：

- 只有干巴巴的命令，没有上下文支撑，导致智能体可能机械执行。
- 为了“极简”而使用了模糊的代词或缩写。
- 借口表（Excuse Smasher）只针对偷懒，没有针对“误解”或“擅作主张”的防御。
- **用户要求“优化所有技能”，你却直接开始输出第一个技能的内容，而没有先产出 `writing-plans` 的计划文档。**
- 计划文档中出现“Step 1: 优化所有技能”这样的聚合步骤。
