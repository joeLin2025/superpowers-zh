# 技能本地化与极致打磨 (Skill Localization & Polishing)

## 概述

此技能旨在将平庸的技能文档转化为**针对中文环境优化的、具有强制约束力的行为准则**。

核心原则：
1.  **从“建议”到“铁律”**：将软性的“应该”转化为硬性的“必须”。
2.  **防御性写作**：预判并封堵智能体偷懒的每一个借口。
3.  **地道中文**：拒绝翻译腔，使用符合中文开发者习惯的专业术语和短句。

## 何时使用

- 创建新的中文技能时。
- 将英文技能翻译并适配到中文环境时。
- 发现现有技能被智能体“绕过”或执行不力时。
- 需要提升技能文档的专业度和约束力时。

## 核心流程

### 1. 结构强制 (Structure Enforcement)

所有技能必须包含以下标准部分，不得遗漏：

| 部分 | 中文标题 | 要求 |
|------|----------|------|
| Frontmatter | (无) | 必须包含 `name` (英文) 和 `description` (中文，以“当...时使用”开头)。 |
| Overview | 概述 | 1-2句话的核心定义 + 核心原则列表。 |
| When to use | 何时使用 | 触发条件的列表（含症状）。明确“何时不使用”。 |
| Core Rules | 核心法则 | **使用主动语态的绝对命令**。禁止“建议”。 |
| Excuses | 借口粉碎机 | 预判智能体的借口，并给出事实反击。 |
| Red Flags | 危险信号 | 触发“停止并重来”的具体行为列表。 |

### 2. 语言升维 (Language Upgrade)

将弱势语言替换为强势语言：

| 弱势语言 (禁止) | 强势语言 (必须) | 理由 |
|----------------|----------------|------|
| “建议使用...” | “**必须**使用...” | 消除选择余地。 |
| “尝试避免...” | “**严禁**...” | 明确界限。 |
| “...有助于...” | (直接删除解释) | 技能是行动指南，不是教科书。 |
| “请确保...” | “验证...” | 强调动作结果。 |
| “如果可能的话...”| (无条件执行) | 移除逃避的条件从句。 |

### 3. 文化适配 (Localization)

- **术语规范**：
    - Commit -> 提交
    - Push -> 推送
    - Merge Conflict -> 合并冲突
    - PR/Pull Request -> PR (通用) 或 拉取请求
    - Bug -> Bug (通用) 或 缺陷
    - Agent -> 智能体
- **句式习惯**：
    - 避免长难句。使用短句。
    - 多用无主语的祈使句（如“执行测试”，而不是“你应该执行测试”）。

## 借口粉碎机 (Excuse Smasher)

| 借口 | 事实反击 |
|------|----------|
| “这个规则太严格了” | 只有严格的规则才能保证一致的质量。 |
| “这只是个小任务，不需要这么复杂” | 习惯是在小任务中养成的。例外是混乱的开始。 |
| “翻译意思对就行了” | 只有地道的表达才能被准确执行，翻译腔会导致误解。 |
| “我以后再补充借口表” | 现在的漏洞现在就得补上。没有借口表的技能是不完整的。 |

## 危险信号 (Red Flags)

如果出现以下情况，**立即停止**并修正技能文档：

- 描述 (Description) 解释了技能“做什么”而不是“何时使用”。
- 正文中出现“建议”、“尽量”、“可能”等模糊词汇。
- 缺少“借口粉碎机”或“危险信号”部分。
- 使用了翻译腔浓重的表达（如“它是为了...”）。

## 示例对比

**Bad (基线):**
> 建议在提交代码前运行测试。这有助于减少 Bug。如果时间不够，可以先跳过。

**Good (打磨后):**
> **提交前必须运行测试。**
>
> **危险信号：** 任何试图跳过测试的行为 -> **立即拒绝提交**。
>
> | 借口 | 事实 |
> |------|------|
> | “时间不够” | 修复 Bug 的时间是运行测试的 10 倍。 |
