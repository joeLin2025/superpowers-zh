# 接收代码审查 (Receiving Code Review)

## 概述

此技能强制执行理性的、基于技术验证的审查反馈处理流程。

**核心原则：**
1.  **先理解后行动**：在动手修改代码之前，必须完全理解审查者的意图。
2.  **技术验证**：不要盲目接受建议。验证它是否真的更好。
3.  **零防御**：将反馈视为改进的机会，而不是对个人的攻击。

## 何时使用

- 收到同事或 AI 对你的代码提出的修改建议时。
- 准备根据反馈修改代码时。
- 对某个反馈感到困惑或不同意时。

## 核心法则

### 1. 必须压制防御本能

- **严禁**立即反驳。即使你觉得对方错了，先深呼吸。
- **严禁**使用防御性语言（“但是...”，“我只是...”）。
- **必须**假设审查者是善意的，并且他们可能看到了你没看到的视角。

### 2. 必须验证技术建议

对于每一条技术建议（如“这里用 map 更好”）：
- **必须**进行思考或实验验证：这真的更简洁/高效/安全吗？
- 如果建议引入了新的 Bug 或复杂性，**必须**有理有据地提出异议。
- **严禁**为了讨好审查者而盲目接受你认为错误的代码。

### 3. 必须分类处理

将反馈分为三类并按顺序处理：
1.  **Blocker (必须修)**：Bug、安全漏洞、严重的设计缺陷。 -> **立即修复。**
2.  **Suggestion (建议修)**：可读性、命名、小优化。 -> **权衡后修复或解释。**
3.  **Question (询问)**：审查者没看懂。 -> **解释并考虑优化代码注释。**

### 4. 严禁过度设计 (YAGNI)

- 如果审查者建议添加一个“未来可能用到”的功能，**必须**礼貌拒绝。
- 引用 YAGNI (You Ain't Gonna Need It) 原则。
- 保持代码的简洁和聚焦。

## 借口粉碎机 (Excuse Smasher)

| 借口 | 事实反击 |
|------|----------|
| “他根本不懂我的代码” | 如果他看不懂，说明你的代码写得不够清晰。代码是写给人看的。 |
| “这个改动太麻烦了” | 技术债务更麻烦。现在的麻烦是为未来省力。 |
| “这只是风格问题” | 风格统一是团队协作的基础。遵循项目的 Lint 规则。 |
| “他是资深工程师，他说的一定对” | 资深工程师也会犯错。你的责任是为代码质量负责，而不是盲从权威。 |

## 危险信号 (Red Flags)

如果出现以下情况，**立即停止**修改：

- 你在没有运行测试的情况下就实施了建议的修改。
- 你为了消除评论而删除了有用的代码。
- 你发现自己正在和审查者进行情绪化的争论。
- 你同意了一个你完全不理解的修改方案。
