# 接收代码审查 (Receiving Code Review)

## 概述

本技能要求以无我心态处理反馈，并通过证据验证完成闭环。

**核心原则**
1. **无我原则**：反馈针对代码，不是针对人。
2. **先理解后行动**：未理解建议前禁止改动。
3. **验证即尊重**：用证据回应反馈。

## 何时使用

- 收到审查意见或反馈建议。

## 强制流程 (Workflow)

### 1. 反馈分类
- **Blocker**：安全、逻辑错误。
- **Structural**：架构/设计调整。
- **Suggestion**：可读性/风格。
- **Question**：理解不清的疑问。

### 2. 证据验证
- 对技术建议必须通过实验或测试验证。
- 若不采纳，必须给出数据或事实依据。

### 3. 闭环确认
- 采纳建议：说明改动与验证。
- 不采纳：说明原因与证据。

## 禁令
- 禁止情绪化回应。
- 禁止无验证执行或拒绝。
- 禁止静默忽略反馈。

## 借口粉碎机 (Excuse Smasher)

| 借口 | 事实反击 |
|------|----------|
| “他不懂我的逻辑” | 如果解释不清，说明设计不够清晰。 |
| “只是风格，不重要” | 风格一致性是维护成本的关键。 |
| “改太麻烦” | 麻烦不是拒绝质量改进的理由。 |

## 危险信号 (Red Flags)

- 反馈未分类直接处理。
- 未验证就执行建议。
- 未说明原因就拒绝建议。
- 情绪化语言出现。
