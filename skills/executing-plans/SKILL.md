# 执行计划 (Executing Plans)

## 概述

本技能将静态计划转换为可控执行流程，强调证据持久化、可靠性检查点与失败熔断，确保执行可追踪、可恢复。

**核心原则**
1. **计划至上**：不得偏离计划执行。
2. **证据物化**：验证输出必须写回计划文件。
3. **检查点停顿**：每个批次结束必须停顿审查。
4. **失败熔断**：任何非预期失败必须停止并回到计划修正。

## 何时使用

- 有 `writing-plans` 生成的计划文件。
- 任务包含多步骤或高风险变更。

## 强制流程 (Workflow)

### 1. 审计与对齐
- **必须**完整加载计划文件。
- **必须**声明“正在执行第 N 批次任务”。
- 若计划存在错误或不可执行步骤，**立即停机**并修正计划。

### 2. 状态同步
- 执行前将任务标记为 `[ ] (进行中)`。
- 执行后标记为 `[x]`。
- **必须**在任务下附上验证日志片段（如测试最后 5 行输出）。

### 3. 批次执行与停顿
- 单批次最多执行 3 个原子任务。
- 批次结束必须停顿，记录审查结论。
- 若交付模式为 `Local Auto-commit`，每步完成必须提交。

### 4. 可靠性检查点
- 在关键步骤后执行最小回归验证。
- 对失败案例必须记录“失败原因 + 影响范围 + 处理动作”。

### 5. 执行熔断
- 测试失败、环境异常或指令模糊：**立即停止**。
- 禁止在失败状态下继续下一步。

## 禁令
- 禁止“顺手修改”计划外内容。
- 禁止仅在聊天中声明验证通过。
- 禁止跳过检查点停顿。

## 借口粉碎机 (Excuse Smasher)

| 借口 | 事实反击 |
|------|----------|
| “改动很小，顺手做了” | 计划外改动不可审计，必须先修订计划。 |
| “日志太长不写” | 没有证据就没有完成，日志是唯一验收凭证。 |
| “先做完再补验证” | 验证是执行的组成部分，不是事后补丁。 |

## 危险信号 (Red Flags)

- 计划文件没有更新但执行已完成。
- 任务缺少验证日志。
- 连续执行超过 3 个任务未停顿。
- 失败后继续下一步。
