# 编写技能 (Writing Skills)

## 概述

本技能以“可执行规范”为目标，通过先红后绿的验证流程保证规则有效。

**核心原则**
1. **先红后绿**：先观察失败，再写规则修正。
2. **可执行性**：规则必须可验证、可审计。
3. **强约束语气**：禁止模糊表达。

## 何时使用

- 新建技能或重构技能。
- 需要固化工程规范时。

## 强制流程 (Workflow)

### 1. 失败基线 (Red)
- 构造一个会触发错误行为的场景。
- 记录失败表现与借口。

### 2. 规则编写 (Green)
- 写出针对失败的硬约束规则。
- 明确触发条件与禁止项。

### 3. 验收标准 (Acceptance)
- 定义可验证的完成标准。
- 设定证据格式（日志、文件路径）。

### 4. 打磨 (Refactor)
- 使用 `skill-localization-and-polishing` 进行信噪比审计。

## 输出模板 (强制)
- Frontmatter（name/description）
- 概述
- 何时使用
- 核心规则
- 借口粉碎机
- 危险信号

## 禁令
- 禁止没有失败基线就写规则。
- 禁止输出“建议/可选”类措辞。
- 禁止缺少验收标准。

## 借口粉碎机 (Excuse Smasher)

| 借口 | 事实反击 |
|------|----------|
| “规则先写出来再说” | 没有失败基线无法证明规则有效。 |
| “不用验收标准” | 无验收标准的规则无法执行。 |
| “用建议语气更友好” | 技能是规则，不是建议。 |

## 危险信号 (Red Flags)

- 没有 Red 阶段记录。
- 没有验收标准。
- 规则含有弱约束词。
